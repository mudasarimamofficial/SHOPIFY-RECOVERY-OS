# BUG-0002: Data Truncation and Order Schema Mismatch

## Context

During a deep forensic trace of the pipeline, three massive structural flaws were identified causing silent data loss in the backup and restore engines.

## The Bugs

1. **Silent Pagination Cap**: `shopify.server.ts` `paged` function had a hardcoded `safety` loop limit of 100. This meant any dataset over 25,000 items (blogs, articles, pages, etc.) would silently truncate.
2. **Order REST/GraphQL Schema Mismatch**: `backup.server.ts` uses GraphQL to backup orders, yielding a node schema (`lineItems`, `displayFinancialStatus`). `executor.ts` tried pushing this raw GraphQL schema into the REST `/orders.json` endpoint, which caused silent failures, dropped line items, and missing ID mappings because the REST endpoint expected `line_items` and `financial_status`.
3. **Article Array Truncation**: `backup.server.ts` had a hardcoded `.slice(0, 20)` on the blogs array, meaning articles from the 21st blog onward were permanently ignored.

## Resolution

1. Raised pagination `safety` limit to `100_000` (25M items) and added an explicit `throw new Error` if the pagination limit is ever exceeded.
2. Re-wrote `restoreOrder` in `executor.ts` to explicitly map the GraphQL Order node into the REST payload format, extracting `variant.id` and `product.id` correctly and applying the `gidMap`.
3. Removed the `.slice(0, 20)` in `backup.server.ts`.
