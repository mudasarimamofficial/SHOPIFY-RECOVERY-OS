# Backup Engine Architecture & Call Graph

## Purpose

The Backup Engine orchestrates the extraction of merchant data from the Shopify Admin API via REST and Bulk Operations, aggregates it into JSONL chunks, and uploads the final compressed payload to the `recovery_packages` Supabase storage bucket alongside a cryptographic checksum.

## Call Graph (End-to-End Execution)

```mermaid
graph TD
  A[User Clicks 'Start Backup'] --> B[Client: startBackup mutation]
  B --> C{Supabase Insert}
  C -->|Creates| D[backups table row]
  C -->|Creates| E[backup_resources tracking rows]
  D --> F[Server: stepBackup()]
  F --> G{Shopify API Poll}
  G -->|If Bulk Required| H[Submit BulkOperation Query]
  G -->|If Small Data| I[Fetch via REST/GraphQL]
  H --> J[Wait/Poll for Bulk Operation URL]
  J --> K[Download JSONL]
  I --> K
  K --> L[Compress to .gz]
  L --> M[Upload to recovery_packages Bucket]
  M --> N[Update backup_resources.bytes / count]
  N --> O{All Resources Complete?}
  O -->|No| F
  O -->|Yes| P[Generate recovery.json Manifest]
  P --> Q[Calculate Checksums]
  Q --> R[Mark Backup Completed]
```

## Failure Modes & Retry Logic

- If the Shopify Admin API returns a `429 Too Many Requests`, the engine uses exponential backoff.
- If a resource fails, the `backup_resources` row status is set to `failed` with the error payload. The engine proceeds to the next resource.

## Known Bugs

- **[BUG-0001] Manifest Version**: The final `recovery.json` is written with `format: "recovery/2"`, which breaks the Restore Engine.
- **Missing DB Manifest**: The engine does not properly save the final JSON manifest into the `backups.manifest` database column.
