# Restore Engine Architecture & Call Graph

## Purpose

The Restore Engine is responsible for downloading a generated `recovery_package` from Supabase Storage, parsing its `recovery.json` manifest, calculating dependency orderings (e.g., Products before Collections), and safely injecting the data into the target Shopify store via the Admin API.

## Call Graph (End-to-End Execution)

```mermaid
graph TD
  A[User Clicks 'Initiate Restore'] --> B[Client: startRestore mutation]
  B --> C[Server: initiateRestore()]
  C --> D{Validate backup_id}
  D --> E[Fetch backups.manifest]
  E --> F{Check manifest version}
  F -->|format !== recovery/1| G[THROW ERROR BUG-0001]
  F -->|Valid| H[generateRestorePlan()]
  H --> I[Map Dependencies]
  I --> J[Insert restore_jobs row]
  J --> K[Server: executeRestoreStep()]
  K --> L[Download Resource JSONL.gz]
  L --> M[Decompress & Parse]
  M --> N[Map Old IDs to New IDs]
  N --> O[Shopify Admin API Create/Update]
  O --> P[Update restore_jobs.report]
  P --> Q{More Steps?}
  Q -->|Yes| K
  Q -->|No| R[Mark Restore Complete]
```

## Dependency Ordering Map

1. **Locations & Inventories**
2. **Products & Variants**
3. **Collections & Custom Collections**
4. **Customers**
5. **Orders**
6. **Metaobjects & Metafields**

## Known Limitations

- The Restore Engine is highly sensitive to the schema changes introduced in `recovery/2` manifests.
- Conflict resolution (what happens if a product already exists) is currently `OVERWRITE` by default, using the Shopify handle as the unique identifier.
